# 부록 2. Evidence Vault 운영 SOP 템플릿

> 이 문서는 등록 후 사용 증거를 모으고, 찾고, 다시 점검하기 위한 실무용 템플릿이다. 그대로 복사해 팀 운영 문서로 써도 된다.

## 1. 목적

이 SOP의 목적은 다음 세 가지다.

- 표장 사용 증거를 한 규칙으로 저장한다.
- 분기별, 반기별 점검에서 누락을 빠르게 찾는다.
- 분쟁, 심사, 내부 검토 때 바로 꺼낼 수 있는 자료를 유지한다.

## 2. 운영 범위

다음 자료를 Evidence Vault에 넣는다.

- 제품 노출 자료
- 판매 화면 자료
- 거래·출고 자료
- 계약·권리 구조 자료
- 플랫폼 운영 자료
- 집행·경고 대응 자료
- 내부 메모와 점검 기록

## 3. 폴더 구조

권장 구조는 단순하게 유지한다. 연도, 분기, 범주 순서로만 나눈다.

```text
Brand_Evidence/
  2026/
    Q1/
      product/
      sales/
      platform/
      contracts/
      enforcement/
      internal-memo/
```

### 폴더 운영 원칙

- 새 자료는 반드시 해당 연도와 분기로 넣는다.
- 범주를 임의로 늘리지 않는다.
- 한 사건의 핵심 자료는 한곳에 모은다.
- 원본과 가공본은 같은 폴더에 섞지 않는다.

## 4. 파일명 규칙

파일명은 아래 순서를 고정한다.

```text
브랜드_채널_YYYY-MM-DD_자료유형_내용_v01
```

### 예시

- `BRANDA_tmall_2026-03-12_product-page_v01.pdf`
- `BRANDA_wechat_2026-03-12_campaign-banner_v01.png`
- `BRANDA_contract_2026-03-12_license-summary_v01.docx`

### 파일명 규칙

- 날짜는 항상 `YYYY-MM-DD`로 쓴다.
- 띄어쓰기는 쓰지 않는다.
- 버전이 바뀌면 `v02`, `v03`처럼 올린다.
- 캡처, 원본, 메모의 성격이 다르면 파일명도 다르게 만든다.

## 5. 증거 카테고리

### 5.1 제품 노출

- 상품 사진
- 라벨
- 패키지
- 사용설명서

### 5.2 판매 노출

- 플랫폼 상세페이지
- 검색 결과 화면
- 장바구니 또는 주문 흐름 화면

### 5.3 거래·출고 자료

- 인보이스
- 발주서
- 출고 내역
- 송장

### 5.4 계약·권리 구조

- 라이선스 계약
- 유통 계약
- 위임 메모
- 권리자 정보

### 5.5 플랫폼 운영

- 캠페인 배너
- 프로모션 페이지
- SNS 게시물
- 전시회 자료

### 5.6 집행·경고 대응

- 경고장
- 신고 기록
- 플랫폼 회신
- 세관 대응 메모

### 5.7 내부 메모

- owner/user 구분 메모
- 점검 기록
- 누락 자료 목록
- 후속 조치 메모

## 6. 메타데이터 항목

각 파일 또는 사건 메모에 아래 정보를 남긴다.

| 항목 | 적는 내용 |
| --- | --- |
| 브랜드명 | 실제 사용 중인 표장 |
| 상품 또는 채널 | 어떤 제품, 어떤 판매 경로인지 |
| 사용 주체 | owner, user, operator 중 누구인지 |
| 수집 날짜 | 자료를 확보한 날짜 |
| 자료 출처 | 플랫폼, 계약서, 촬영, 내부 전달 등 |
| 관련 사건번호 | 이의, 경고, 신고, 대응 번호 |
| 보관 위치 | 폴더 경로와 파일명 |
| 다음 점검일 | 다음 분기 또는 반기 점검 날짜 |

## 7. 입력 절차

1. 자료를 수집한다.
2. 중복 파일을 제거한다.
3. 범주를 정한다.
4. 날짜를 맞춘다.
5. owner/user 연결을 표시한다.
6. 폴더에 저장한다.
7. 사건 메모를 갱신한다.

### 저장 전 확인

- 표장이 실제로 보이는가
- 날짜가 남는가
- 어떤 상품인지 바로 읽히는가
- 누가 사용했는지 연결되는가

## 8. 분기별 점검 질문

분기 점검은 현재도 쓰고 있는지 확인하는 단계다.

- 핵심 표장이 실제 판매 화면에 남아 있는가
- 중국어 표기가 등록 내용과 다른가
- 새 SKU가 기존 범위를 벗어났는가
- 사용자 구조가 바뀌었는가
- 최근 분기 자료가 충분히 쌓였는가
- 누락된 채널이 있는가

### 분기 점검 산출물

- 1페이지 점검 메모
- 누락 자료 목록
- 보완이 필요한 채널 목록
- 다음 분기 행동 항목

## 9. 반기별 점검 질문

반기 점검은 운영 구조 자체를 다시 맞추는 단계다.

- 권리자 정보가 현재 법인 구조와 맞는가
- 대리인 연락망이 살아 있는가
- 라이선스와 실제 운영이 일치하는가
- 집행 이력과 경고 메모가 한곳에 모여 있는가
- 새 지역, 새 채널, 새 상품군이 기존 포트폴리오와 충돌하지 않는가

### 반기 점검 결과

- 유지: 구조와 증거가 정상
- 보완: 자료는 있으나 정리가 부족
- 재설계: 사용자 구조나 채널 구조를 다시 짜야 함

## 10. 자주 하는 실수

### 10.1 화면만 모으고 거래 흔적을 안 남김

보여주기 좋은 캡처는 많지만, 실제 판매나 출고를 보여주는 자료가 없다.

### 10.2 owner와 user를 섞어 씀

등록 주체와 실제 사용 주체가 다른데 설명 메모가 없다.

### 10.3 중국어 표기가 계속 바뀜

상세페이지, 포장, 계약서, 플랫폼에서 이름이 서로 다르다.

### 10.4 새 SKU를 나중에 생각함

상품팀이 먼저 출시하고 IP팀은 뒤늦게 본다.

### 10.5 폴더는 있는데 못 찾음

파일명과 날짜 규칙이 없어서 필요한 날에 다시 찾지 못한다.

## 11. 운영 체크리스트

- [ ] owner/user 구조를 적었다
- [ ] 핵심 표장 캡처를 저장했다
- [ ] 제품·패키지 증거를 넣었다
- [ ] 거래 자료를 넣었다
- [ ] 플랫폼 자료를 넣었다
- [ ] 계약 자료를 넣었다
- [ ] 사건 메모를 업데이트했다
- [ ] 다음 점검일을 적었다

## 12. 짧은 운영 원칙

- 적는 것보다 찾는 데 초점을 둔다.
- 한 사건은 한 메모로 남긴다.
- 새 자료는 같은 규칙으로만 넣는다.
- 점검일이 지나면 반드시 다시 본다.
- 기록은 늦게 하면 무너진다.
