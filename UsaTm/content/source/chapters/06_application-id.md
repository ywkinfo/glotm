# 출원서 작성 실무와 ID 설계

출원서 품질은 입력값에서 결정된다. owner와 user가 맞는지, 표장 버전이 고정됐는지, goods/services가 실제 사업과 specimen에 맞는지 여기서 갈린다. 이 단계가 흔들리면 심사에서만 끝나지 않고 유지관리와 enforcement까지 같이 무너진다.

## 1. 왜 이 단계가 중요한가

출원서는 권리의 설계도다. USPTO는 "무슨 브랜드를 누가 어떤 범위로 쓸 것인지"를 서류로 본다. 그래서 입력 오류는 단순 오타가 아니다. 잘못된 owner, 잘못된 표장 버전, 과도하거나 빈약한 ID는 나중에 수정 비용으로 돌아온다.

실무에서는 세 가지가 가장 자주 꼬인다.

- 브랜드 운영법인과 출원인 법인이 다르다.
- 실제 쓰는 표장과 출원하는 표장 버전이 다르다.
- goods/services는 넓게 적었는데 specimen은 그 범위를 못 받친다.

출원서 작성은 법무팀의 입력 작업이 아니라, 사업 구조와 증거 구조를 한 장에 맞추는 작업이다.

## 2. owner와 entity를 먼저 맞춘다

가장 먼저 정할 것은 "누가 출원인이냐"가 아니다. "누가 이 표장을 실제로 통제하고, 누가 사용을 입증할 수 있느냐"다. 미국에서는 owner-user 정합성이 깨지면 이후 assignment, license, specimen, 유지제출에서 계속 흔적이 남는다.

### 먼저 확인할 항목

- 한국 본사, 미국 자회사, 유통사, licensee 중 실제 owner가 누구인지
- 미국 내 실제 판매 주체와 owner가 같은지
- license 구조가 있다면 quality control이 문서와 운영에 같이 있는지
- 향후 리브랜딩이나 법인 재편 시 owner chain이 흔들리지 않는지

owner는 계약서 한 줄로 끝나지 않는다. 출원서, specimen, invoice, 웹사이트 하단 표기, 내부 승인 문서가 같은 방향을 가리켜야 한다. 이게 안 맞으면 상대방이 공격할 포인트를 스스로 만든다.

### 자주 나는 실수

| 오류 | 문제 | 실무상 결과 |
|---|---|---|
| 본사 브랜드인데 미국 판매법인으로 출원 | 사용 통제와 권리귀속이 갈라짐 | assignment 정리와 증거 설명이 꼬임 |
| 라이선스만 있고 quality control 기록이 없음 | naked license 의심 | 권리 약화 리스크 |
| 법인명은 맞지만 주소·국가 정보가 어긋남 | 데이터 무결성 저하 | 심사 대응 때 같은 오류가 반복됨 |
| 실제 운영 주체와 다른 명의로 출원 | specimen과 사용 사실 설명이 어려움 | 거절이나 보정 부담 증가 |

## 3. mark version을 먼저 고정한다

표장 버전은 출원서의 뼈대다. 문자표장, stylized mark, composite mark 중 무엇을 보호할지 먼저 정해야 한다. 여기서 흔들리면 나중에 같은 브랜드인데도 다른 권리 묶음이 생긴다.

### 버전 선택의 기준

- **문자표장**: 단어 자체의 보호가 필요할 때
- **스타일화 표장**: 특정 디자인, 글꼴, 배열이 핵심일 때
- **복합표장**: 단어와 로고를 같이 쓰는 실제 사용형태가 중요할 때

선택 기준은 멋이 아니다. 실제 사용 화면과 포장에 무엇이 고정돼 있는지다. 브랜드팀이 슬로건을 자주 바꾸는데 법무가 하나의 버전으로 밀어붙이면 specimen 단계에서 다시 맞춰야 한다.

### 이때 보는 것

- 웹사이트, 패키지, 앱 화면에서 실제로 보이는 버전
- 단어만 남겨도 의미가 살아 있는지
- 로고 요소가 바뀌어도 핵심 식별력이 유지되는지
- 해외 지역별 사용본이 너무 달라지지 않는지

같은 브랜드라도 버전이 달라지면 다른 권리처럼 취급될 수 있다. 출원 전에 "이번 건은 단어를 가져가는 건지, 디자인까지 묶는 건지"를 한 문장으로 말할 수 있어야 한다.

## 4. goods/services ID는 사업을 그대로 옮기지 않는다

ID는 사업계획서 복사본이 아니다. 그렇다고 너무 법률문구처럼 적는 것도 아니다. 목적은 간단하다. 실제 사용과 가까운 범위에서, 나중에 증거로 설명 가능한 수준으로 적는 것이다.

### 좋은 ID의 기준

- 실제 판매품이나 제공 서비스와 연결된다
- specimen으로 설명할 수 있다
- core business와 adjacent line을 구분한다
- 너무 넓지도, 너무 좁지도 않다

너무 넓으면 나중에 사용 입증이 약해진다. 너무 좁으면 사업 확장 때 새 출원이 계속 생긴다. 미국 실무에서는 이 둘의 균형이 핵심이다.

### core vs adjacent coverage

core coverage는 지금 매출을 만드는 상품과 서비스다. adjacent coverage는 바로 옆에 붙은 확장 영역이다. 둘을 섞어 쓰면 포트폴리오가 뚱뚱해진다. 반대로 core만 적으면 사업이 커질 때 다시 출원해야 한다.

실무적으로는 이렇게 본다.

- **core**: 현재 판매, 현재 청구, 현재 웹사이트에서 바로 보이는 항목
- **adjacent**: 12~24개월 안에 실제로 출시될 가능성이 높은 인접 항목
- **do not file yet**: 아직 제품 정의도 없고 specimen도 없는 먼 미래 항목

adjacent까지 잡을 수 있으면 좋지만, 그 문구를 실제 사업과 연결할 수 있어야 한다. 그냥 "언젠가 할지도 모른다"는 이유로 넣는 건 좋지 않다.

### ID 설계 원칙

1. 지금 쓰는 말을 먼저 적는다.
2. 그 말을 USPTO 문구로 정리한다.
3. specimen으로 설명되는지 확인한다.
4. core와 adjacent를 분리한다.
5. 필요하면 class를 나눈다.

이 순서를 거꾸로 하면 보통 넓고 허술한 문구가 나온다. 법률적으로는 그럴듯해 보여도 운영에는 별 도움이 안 된다.

## 5. specimen은 나중에 붙이는 서류가 아니다

specimen은 출원문구를 검증하는 시험지다. 그래서 goods/services를 적을 때부터 specimen 가능성을 같이 봐야 한다. 실제 포장, 상품 페이지, 서비스 화면에서 설명 가능한 문구가 아니면 나중에 다시 흔들린다.

### specimen 정합성 체크

- 표장 버전이 출원서와 같은가
- goods/services가 실제 화면이나 포장에 드러나는가
- 소비자가 그 화면만 보고 제품이나 서비스를 이해할 수 있는가
- 내부 시안이 아니라 실제 상거래 자료인가

특히 서비스는 많이 틀린다. 랜딩페이지가 있긴 한데 서비스 설명이 너무 얇거나, 장식용 브랜드 문구만 있고 서비스 제공 사실이 안 보이면 specimen 설명이 약해진다.

### 자주 망가지는 조합

- 출원은 문자표장인데 specimen은 로고 중심 화면
- goods/services는 넓은데 실제 페이지는 하나의 제품만 보여줌
- 미국용 표장과 해외용 표장이 달라 동일성 설명이 어려움
- 앱 스크린샷은 있는데 실제 제공 서비스가 드러나지 않음

출원문구와 specimen은 따로 노는 두 문서가 아니다. 같은 이야기를 다른 방식으로 증명해야 한다.

## 6. multi-class는 많이 넣는 게 답이 아니다

class를 늘리면 보호 범위가 커 보인다. 하지만 관리 비용도 같이 오른다. multi-class는 포트폴리오를 키우는 도구이지, 넓어 보이는 것 자체가 목표가 되어서는 안 된다.

### class를 나눌지 볼 때

- 실제로 다른 사업인가
- specimen과 사용 시점이 다르게 움직이는가
- 유지관리와 audit 부담이 커지는가
- 같은 브랜드를 다른 팀이 따로 운영하는가

한 출원에 다 넣으면 편해 보이지만, 나중에 사용 입증과 유지관리에서 더 복잡해질 수 있다. 반대로 너무 쪼개면 중복 비용과 내부 관리가 늘어난다. 그래서 class는 법률 기준만이 아니라 운영 기준도 함께 보고 나눠야 한다.

### 실무 판단

| 상황 | 권장 방향 |
|---|---|
| 현재 제품군이 명확하고 확장도 비슷한 라인 | 핵심 class 중심으로 설계 |
| 제품과 서비스가 완전히 다름 | class 분리 검토 |
| 일부만 실제 출시, 나머지는 기획 단계 | core만 먼저 출원 |
| 같은 브랜드 아래 라인이 여러 개지만 증거가 다름 | 각 line별 specimen 가능성 점검 |

멀티클래스는 "많이"보다 "설명 가능"이 중요하다. 관리 못 할 범위를 넣는 건 미래의 문제를 오늘로 당겨오는 일이다.

## 7. 흔한 데이터 입력 실패

출원서 실패의 상당수는 법리가 아니라 데이터다. 아래 항목은 출원 직전 반드시 다시 봐야 한다.

### 흔한 실패 패턴

- owner name이 내부 문서와 다름
- 주소 형식이 출원인 유형과 안 맞음
- 표장 버전 파일이 오래된 것임
- goods/services를 한국식 문장으로 직역함
- class 선택은 맞는데 문구가 실제 사업과 안 맞음
- specimen 후보가 없는 ID를 넣음
- 통지 담당자가 정해져 있지 않음

### 방지법

- 출원 마스터 시트를 먼저 만든다
- 법인명, 주소, 사용자, 표장 버전, class, specimen 후보를 한 줄로 묶는다
- 내부 승인본과 출원 입력본을 분리하지 않는다
- 출원 후 담당자 변경이 생기면 즉시 기록을 업데이트한다

입력 데이터는 한번 틀리면 그 오차가 계속 복제된다. 그래서 출원서 작성은 "한 번 더 확인"이 아니라, 같은 정보를 서로 다른 문서끼리 대조하는 일이다.

## 8. 제출 전 최종 QC

출원 직전에는 아래 순서로 본다. 이건 실제 제출 체크리스트로 써도 된다.

### pre-filing QC checklist

- [ ] owner가 실제 사용 주체와 맞는다
- [ ] 법인명, 주소, 국가 정보가 내부 승인본과 일치한다
- [ ] 표장 버전이 최종본으로 고정됐다
- [ ] 이번 건이 문자표장, stylized mark, composite mark 중 무엇인지 정리됐다
- [ ] goods/services가 현재 사업과 연결된다
- [ ] core coverage와 adjacent coverage가 분리돼 있다
- [ ] 넣지 말아야 할 미래형 항목을 걸러냈다
- [ ] specimen 후보가 실제로 존재한다
- [ ] specimen이 출원 표장 버전과 맞는다
- [ ] class를 나눈 이유를 설명할 수 있다
- [ ] multi-class가 정말 필요한지 다시 봤다
- [ ] 통지 수령과 내부 담당자 체계가 정해졌다
- [ ] 입력본, 첨부본, 승인본 사이에 충돌이 없다

마지막 질문은 단순하다. "이 출원서를 보고 나중에 누가, 어떤 표장을, 어떤 범위로, 어떤 증거로 쓰는지 설명할 수 있나." 답이 흐리면 아직 제출할 때가 아니다.

## 9. 한 줄 정리

좋은 출원서는 넓은 문구가 아니라 맞는 문구다. owner, mark version, ID, specimen이 한 방향으로 정렬되면 심사도, 유지도, enforcement도 훨씬 단단해진다.
