# 사용 개시, Specimen, Statement of Use 운영

specimen은 미국 연방 상표에서 “우리가 진짜로 쓰고 있다”를 보여주는 핵심 증거다. intent-to-use로 가면 출원보다 더 중요한 구간이 뒤에 온다. 실제 사용, 증거 수집, 버전 통제, 제출 판단이 한 덩어리로 움직여야 한다.

## 1. 이 장이 다루는 범위

이 장의 목적은 단순한 정의 설명이 아니다. 무엇이 좋은 specimen인지, 언제 Statement of Use를 준비해야 하는지, 누가 무엇을 보관해야 하는지, 그리고 어떤 실패를 막아야 하는지를 운영 관점에서 정리한다.

특히 중요한 전제는 세 가지다.

- specimen은 디자인 파일이 아니라 상거래 증거다.
- intent-to-use는 “나중에 하자”가 아니라 launch plan과 docket plan을 동시에 요구한다.
- 제출 순간보다 제출 전 준비가 더 중요하다.

## 2. product specimen과 service specimen은 다르게 본다

제품과 서비스는 증거의 모양이 다르다. 같은 로고라도 무엇을 증명하려는지에 따라 읽히는 방식이 완전히 달라진다.

### 제품용 specimen

제품에서는 표장이 실제 상품과 연결되어 보여야 한다. 일반적으로 아래 자료가 강하다.

- 제품 포장과 라벨
- 행태그, 스티커, 박스, 인서트
- 제품 자체에 부착된 표기
- 실제 판매 화면에서 상품 이미지와 함께 표시된 상표
- 출고나 판매 현황을 설명하는 화면 중 상품과 표장이 함께 읽히는 자료

핵심은 “이 표장이 이 상품을 가리킨다”는 연결이다. 로고만 단독으로 제시된 자료는 약하다.

### 서비스용 specimen

서비스에서는 물건처럼 만질 수 있는 포장이 없으므로, 서비스가 실제로 제공된다는 맥락이 보여야 한다.

- 서비스 소개 웹페이지
- 예약, 상담, 견적, 가입, 신청 화면
- 공개 배포되는 브로슈어, 전단, PDF 소개서
- 앱이나 플랫폼 안에서 서비스가 설명되는 화면
- 서비스 제공 사실이 읽히는 광고 페이지

여기서도 핵심은 동일하다. 표장이 서비스 이름으로 기능하고, 사용자가 그 서비스를 이해할 수 있어야 한다.

브로슈어나 PDF 소개서도 서비스 specimen이 될 수는 있지만, 공개 맥락과 서비스 설명이 읽혀야 한다. 내부 제안서나 영업 초안처럼 외부 사용자가 접하지 않는 문서는 보통 약하다.

### product vs service 판단 기준

| 질문 | 제품 | 서비스 |
|---|---|---|
| 사용자가 무엇을 사나 | 물리적 상품 | 제공 행위, 접근권, 기능 |
| 강한 증거는 무엇인가 | 포장, 라벨, 실제 상품 사진 | 서비스 설명 페이지, 신청 화면 |
| 약한 증거는 무엇인가 | 로고 파일, 내부 목업 | 내부 소개서, 비공개 슬라이드 |
| 가장 흔한 실수는 무엇인가 | 상품과 표장 연결이 끊김 | 서비스 제공 맥락이 없음 |

## 3. 강한 증거와 약한 증거

좋은 specimen은 시각적으로 깔끔한 자료보다 검토를 거쳐도 설명 가능한 자료에 가깝다. 아래처럼 보면 판단이 쉽다.

| 일반적으로 강한 편인 증거 | 이유 |
|---|---|
| 실제 판매 중인 제품 포장에 표장이 붙은 사진 | 상거래 현장과 직접 연결된다 |
| 상품 이미지, 제품명, 주문 동작이 함께 보이는 판매 화면 | 사용과 판매 의도가 동시에 읽힌다 |
| 서비스명, 서비스 설명, 신청 동작이 있는 공개 웹페이지 | 서비스 제공 사실이 드러난다 |
| 외부 소비자가 볼 수 있는 광고나 브로슈어 | 내부 문서가 아니라 외부 노출 자료다 |

| 일반적으로 약한 편인 증거 | 이유 |
|---|---|
| 내부 시안, 목업, 제안서 | 실제 사용이 아니다 |
| 로고만 있는 파일 | 상품이나 서비스와 연결되지 않는다 |
| 판매 정보 없는 화면 캡처 | 무엇이 팔리는지 안 보인다 |
| 추후 발행 예정인 디자인 이미지 | 미래 계획이지 현재 사용이 아니다 |

실무에서 자주 생기는 문제는 “이건 거의 출시됐다”는 판단에 기대는 것이다. 하지만 제출 단계에서는 실제 사용과 공개 맥락이 확인되는 자료가 더 중요하다.

### 웹페이지 specimen의 형식 요건

웹페이지나 앱 기반 화면을 specimen으로 쓸 때는 화면 내용만 맞다고 끝나지 않는다. URL과 접근일(또는 출력일)까지 함께 남겨야 한다. 캡처 이미지 자체에 표시하든, 출력본에 넣든, 제출 양식에서 함께 제공하든 나중에 다시 설명 가능한 형태여야 한다.

또한 목업, 인쇄 예정 시안, 비공개 테스트 화면은 실제 사용 자료로 보기 어렵다. 공개된 상거래 맥락과 날짜 정보가 함께 남아야 제출 품질이 안정적이다.

## 4. Notice of Allowance 이후 운영

NOA가 나오면 끝이 아니라 시작이다. 이 시점에서 해야 할 일은 하나다. 실제 사용으로 전환할 수 있는지, 아니면 아직 준비가 덜 됐는지를 냉정하게 판단하는 것이다.

### NOA 직후 해야 할 일

1. docket에 NOA와 내부 검토일을 즉시 입력한다.
2. 해당 표장의 최종 사용 버전을 고정한다.
3. owner, user, 실제 판매 주체를 다시 확인한다.
4. goods/services별로 필요한 specimen 유형을 나눈다.
5. 마케팅, 운영, 이커머스, 법무가 같은 상태표를 보게 한다.
6. SOU 제출 가능 여부를 먼저 판단하고, 불가하면 다음 조치를 정한다.

### SOU를 바로 준비할 때

아래 조건이 맞으면 제출 준비가 빠르다.

- 실제 사용이 이미 시작됐다.
- 사용한 표장이 출원 표장과 일치한다.
- class와 goods/services 범위가 증거와 맞는다.
- 외부 노출 화면이나 포장 사진을 바로 확보할 수 있다.
- 내부 검토자가 하나의 최종본을 승인할 수 있다.

### 아직 준비가 안 됐을 때

아직 런칭이 불완전하면 억지로 맞추지 말고 이유를 문서화해야 한다.

- 출시 일정이 늦어짐
- 패키지 인쇄가 지연됨
- 웹페이지가 비공개 상태임
- 표장 버전이 아직 확정되지 않음
- 실제 사용 주체가 정리되지 않음

이때 중요한 것은 “조금만 더 기다리면 되겠지”를 경계하는 일이다. NOA 이후 운영은 감이 아니라 상태표로 관리해야 한다.

## 5. Statement of Use readiness 판단

Statement of Use는 형식 서류가 아니라 사용의 현실성을 걸러내는 관문이다. 제출 전에 아래 질문에 답할 수 있어야 한다.

- 누가 실제 사용주체인가
- 어떤 goods/services가 현재 사용 중인가
- 표장이 출원과 같은 버전인가
- 외부 소비자가 볼 수 있는 증거인가
- 제출 패키지가 나중에 다시 꺼내도 설명 가능한가

### 제출 가능 신호

- 제품 사진과 판매 화면이 서로 맞는다
- 서비스 소개와 신청 화면이 같은 문맥 안에 있다
- 증거 날짜가 현재 사용 시점과 가깝다
- 파일명이 제각각이 아니라 하나의 체계로 정리돼 있다
- 법무 검토자가 “이건 설명 가능하다”라고 말할 수 있다

### 제출하면 안 되는 신호

- 목업만 있고 실물 사진이 없다
- 내부 검토용 슬라이드만 있다
- 서비스는 준비 중인데 공개 화면이 없다
- 표장 버전이 여러 개 섞여 있다
- 누가 최종 승인자인지 모호하다

## 6. evidence storage SOP

specimen 관리에서 제일 중요한 건 저장 방식이다. 증거는 나중에 다시 찾아 설명할 수 있어야 의미가 있다.

### 저장 원칙

- 원본 파일을 보관한다.
- 제출용 PDF나 이미지와 원본을 분리한다.
- 파일명에 날짜, 제품군, 채널, 표장 버전을 넣는다.
- 한 건당 한 폴더, 한 버전 원칙을 둔다.
- 누가 올렸고 누가 승인했는지 남긴다.

### 추천 폴더 구조

`/tm/usatm/mark-name/class-xx/use-evidence/yyyymm/`

위 경로는 하나의 예시일 뿐이다. 중요한 것은 보기 좋게 정리하는 것보다, 6개월 뒤 다른 사람이 봐도 바로 꺼낼 수 있는 규칙을 만드는 일이다.

### 보관 메타데이터

최소한 아래 정보는 같이 남겨야 한다.

- 표장명
- 제품 또는 서비스명
- 사용 국가와 채널
- 캡처 날짜 또는 촬영 날짜
- 공개 URL 또는 출처
- owner / user
- 승인자
- 제출 여부

### SOP에서 빠지면 안 되는 것

- 제출본과 원본을 섞지 않는다.
- 오래된 버전은 덮어쓰지 않는다.
- 화면 캡처만 믿지 말고 설명 메모를 붙인다.
- 패키지 변경 시 예전 버전도 같이 보관한다.

증거 저장 구조가 정리되지 않으면, 나중에 중요한 파일을 찾는 시간이 길어진다. 특히 담당자가 바뀌거나 제출 시점이 늦어질수록 이런 문제가 커진다.

## 7. 팀 책임 분장

specimen 업무는 법무만의 일이 아니다. 실제로는 여러 팀이 한 번씩은 손을 대야 한다.

| 역할 | 책임 |
|---|---|
| 브랜드팀 | 최종 표장 버전 확정, 리브랜딩 일정 통제 |
| 마케팅 | 공개 웹페이지, 광고, 브로슈어, 런칭 자료 확보 |
| 이커머스/운영 | 상품 상세, 주문 화면, 패키지, 출고 자료 확보 |
| 현지팀 | 현장 사용 증거, 판매 화면, 유통 자료 수집 |
| 법무/IP | specimen 적합성 판단, SOU 제출 여부 결정 |
| 외부 counsel | 제출 패키지 검토, 리스크 확인, 파일링 지원 |

가장 중요한 원칙은 담당 공백이 생기지 않게 하는 것이다.

## 8. 흔한 실패 패턴

### 1) 목업만 있다

디자인 시안은 많지만 실제 사용이 없다. 이건 가장 전형적인 실패다. 제품이든 서비스든 외부 노출이 있어야 한다.

### 2) 표장 버전이 뒤섞인다

본사는 워드마크를 쓰고, 현지팀은 축약형을 쓰고, 마케팅은 새로운 로고를 쓴다. 그러면 specimen이 서로 다른 브랜드처럼 보인다.

### 3) goods/services보다 범위가 넓다

실제로는 한 품목만 팔았는데 지정 범위 전체를 대표한다고 생각한다. 나중에 설명이 약해진다.

### 4) 날짜가 맞지 않는다

캡처 시점이 너무 늦거나 너무 앞선다. 현재 사용을 보여주지 못하면 의미가 약해진다.

### 5) 저장이 늦다

런칭 후 몇 달이 지나서 모으기 시작하면 파일이 흩어진다. 담당자도 바뀌고, URL도 바뀌고, 기억도 흐려진다.

### 6) 법무만 알고 있다

마케팅과 운영팀이 specimen 요구를 모르면, 실제 파일이 제때 수집되지 않는 경우가 많다. 실무에서는 이 문제가 가장 자주 반복된다.

## 9. 운영 체크리스트

다음 질문에 대부분 “예”라고 답할 수 있으면 Statement of Use 준비가 비교적 안정적으로 진행되고 있다고 볼 수 있다.

- 현재 사용 중인 표장이 하나로 고정돼 있는가
- 제품 또는 서비스와 표장이 실제로 연결되는가
- 외부 소비자가 볼 수 있는 자료가 있는가
- goods/services와 증거가 서로 맞는가
- owner와 실제 사용주체가 정리돼 있는가
- 최종 제출본과 원본이 분리 보관되는가
- 파일명과 폴더 구조가 일관적인가
- 법무, 브랜드, 운영이 같은 상태표를 보는가
- 누가 승인하고 누가 제출하는지 정해졌는가
- 제출 후에도 다시 꺼내 설명할 수 있는가

## 10. 마무리

좋은 specimen은 멋진 이미지가 아니라 실제 사용의 흔적이다. Statement of Use는 그 흔적을 제출 가능한 구조로 정리하는 마지막 단계다.

운영의 기준은 비교적 분명하다. 제품이면 포장과 판매 화면, 서비스면 제공 화면과 신청 흐름, 그리고 둘 다 공통으로 버전 통제와 저장 SOP가 있어야 한다. 이 기준이 맞으면 SOU는 단순한 제출 서류가 아니라 일하는 시스템에 가까워진다.
