# 글로벌 사용 증거 수집 운영 시스템 구축

브랜드가 여러 국가에 동시에 진입하기 시작하면, 사용 증거는 분쟁이 생긴 뒤에 급하게 모으는 자료가 아니라 평소 운영 시스템의 일부가 됩니다. 실제 문제는 자료가 없는 것보다, 필요한 순간에 꺼내 쓸 수 있는 형태로 정리돼 있지 않다는 데서 더 자주 발생합니다.

실무에서 더 자주 벌어지는 실패는 "자료를 찾지 못했다"가 아니라 "48시간 안에 제출 가능한 증거 패킷으로 묶지 못했다"는 상황입니다. 판매 화면은 마케팅팀 개인 폴더에 있고, 송장과 주문 데이터는 운영팀 ERP에 있고, 패키지 변경 이력은 외주 디자이너 메일함에만 남아 있으면, 분쟁이나 플랫폼 신고가 들어왔을 때 하나의 신뢰할 수 있는 묶음으로 정리하기 어렵습니다.

이 리포트는 미국, 중국, 멕시코처럼 실무 요구가 자주 달라지는 국가를 함께 염두에 두고, 어떤 자료를 누가 어떤 이름으로 보관하고 언제 점검해야 하는지 공개용 실무 가이드 수준에서 정리합니다. 목표는 거대한 시스템을 만드는 것이 아니라, 누가 담당해도 같은 방식으로 계속 쌓이는 운영 구조를 만드는 것입니다.

### 왜 지금 사용 증거 운영 시스템이 필요한가

출원은 한 번으로 끝나도, 사용 증거는 시장이 커질수록 계속 쌓여야 합니다. 문제는 시간이 갈수록 복구가 더 어려워진다는 점입니다. 담당자가 바뀌면 파일 위치를 아무도 모르고, 플랫폼 화면이 교체되면 예전 판매 흔적이 사라지고, 유통 파트너는 과거 판매 기록을 오래 보관하지 않는 경우가 많습니다.

#### 가장 자주 실패하는 장면

많은 팀이 "나중에 필요하면 모은다"는 방식으로 움직입니다. 이 방식은 평소에는 조용하지만, 사건이 생기면 바로 무너집니다.

- 판매 페이지는 있는데 실제 주문·배송 자료가 없어 거래 흐름을 설명하지 못한다.
- 주문 자료는 있는데 어떤 표장이 어떤 상품에 붙어 팔렸는지 보여줄 화면이 없다.
- 포장 이미지는 있는데 그 표기를 누가 승인했고 현재 등록 표기와 어떤 관계인지 설명 메모가 없다.
- 계열사나 유통사가 실제 판매했는데 권리자와 사용자의 연결 문서가 없어 "누구의 사용인가"에서 막힌다.

#### 국가별 요건보다 먼저 갖춰야 할 공통 구조

국가별 요건 차이를 먼저 외우기보다, 공통으로 재사용 가능한 증거 구조를 먼저 설계해야 합니다. 어느 국가든 결국 확인하려는 것은 아래 다섯 가지입니다.

- **거래 흔적**: 실제로 팔렸는가. 주문, 송장, 결제 기록, 배송 완료 자료가 여기에 들어갑니다.
- **상품·서비스 표시**: 어떤 형태로 표기되었는가. 제품명, 로고, 라벨, 상세페이지 표기가 여기에 해당합니다.
- **실제 사용 화면**: 소비자에게 어떻게 보였는가. 판매 페이지, 앱 화면, 광고 집행 화면이 여기에 포함됩니다.
- **시점 증명**: 언제 사용했는가. 캡처일, 메타데이터, 집행 기간, 주문일이 핵심입니다.
- **권리자와 사용자의 연결**: 등록 명의자와 실제 판매 주체가 어떻게 연결되는가. 계열사 관계, 라이선스, 유통계약, 내부 승인 기록이 필요합니다.

이 다섯 가지가 같은 폴더 체계와 같은 명명 규칙 안에서 이어져 있어야 국가별 대응이 빨라집니다. 국가별 차이는 이 공통 구조 위에서 강조점만 조정하는 문제에 가깝습니다.

#### 증거 패킷의 최소 단위

증거는 파일 몇 개를 한 폴더에 넣는다고 바로 제출 패킷이 되지 않습니다. 아래 다섯 가지가 한 묶음으로 연결돼야 실무에서 바로 쓸 수 있습니다.

| 항목 | 최소 예시 | 빠지면 생기는 문제 |
|---|---|---|
| 판매 화면 | 상품 상세페이지, 마켓플레이스 리스팅 캡처 | 무엇이 어떤 표장으로 소비자에게 보였는지 설명이 약해진다 |
| 거래 증빙 | 주문 확인서, 송장, 결제·배송 기록 | 실제 상업적 사용인지 설득력이 떨어진다 |
| 표기·패키지 자료 | 라벨, 포장, 제품 실물 사진 | 등록 표기와 실제 사용 표기의 연결이 약해진다 |
| 날짜 증빙 | 캡처일, 메타데이터, 캠페인 집행 기간 | 사용 시점을 특정하지 못한다 |
| owner-user linkage 메모 | 권리자, 실제 사용자, 계약·계열사 연결 설명 | "누구의 사용인가"에서 설명이 멈춘다 |

특히 마지막 항목은 자주 빠집니다. 권리자는 본사인데 실제 판매는 현지 법인이나 유통사가 하는 구조라면, 자료가 많아도 설명 메모가 없으면 제출 속도가 급격히 떨어집니다.

### 최소 운영 구조

사용 증거 운영 체계는 거대한 데이터 플랫폼이 아니라, 반복 가능한 폴더 구조와 책임 배분에서 시작할 수 있습니다. 핵심은 누가 보더라도 같은 위치에서 같은 자료를 찾을 수 있게 만드는 것입니다.

#### 폴더 구조와 시장별 index 파일

시장별로 폴더를 나누되, 각 폴더 아래에 같은 순서의 하위 폴더와 1페이지짜리 index 파일을 둡니다.

```text
/UsageEvidence
  /{시장코드}_{브랜드명}
    /00_Index
    /01_Sales
    /02_Transaction
    /03_Packaging
    /04_Advertising
    /05_Internal
    /06_ThirdParty
```

예시: `/UsageEvidence/KR_BrandA/01_Sales/`

`00_Index`에는 아래 정도의 관리표를 두는 편이 좋습니다.

| 항목 | 적어둘 내용 |
|---|---|
| Owner | 시장 담당자, 법무 검수 담당자 |
| 최근 업데이트일 | 마지막 수집·점검 날짜 |
| 주력 채널 | 자사몰, Amazon, Naver Shopping, TikTok Shop 등 |
| 관련 계약 위치 | 라이선스, 유통계약, 계열사 사용 승인 문서 위치 |
| 비고 | 표기 변경, 채널 변경, 보완 필요 메모 |

이 index 파일이 있으면 폴더가 단순 저장소가 아니라 운영 단위로 바뀝니다. 누가 책임지는지, 어디가 비었는지, 어떤 계약을 같이 봐야 하는지가 한 장에서 보이기 때문입니다.

#### 파일명 규칙과 필수 메타데이터

파일명은 검색성과 시간순 정렬을 위해 최대한 단순하게 고정하는 편이 좋습니다.

`{수집일자}_{시장코드}_{브랜드명}_{채널}_{내용설명}.{확장자}`

예시: `20250315_KR_BrandA_NaverShopping_ProductPage.png`

다만 파일명만 맞으면 충분하다고 보면 곤란합니다. 아래 메타데이터를 함께 남겨야 나중에 설명이 붙습니다.

- URL 또는 원문 출처
- 캡처일 또는 수집일
- 캡처자 또는 수집자
- 주문번호, 송장번호, SKU 중 연결 가능한 식별자
- 등록 표기 버전과 실제 사용 표기 버전
- 원본 파일과 제출용 편집본의 구분

실무적으로는 캡처 이미지 옆에 같은 이름의 메모 파일을 두거나, index 파일에 행 단위로 관리하는 방식이 가장 유지하기 쉽습니다. 중요한 것은 "이 파일이 무엇인지"가 아니라 "어떤 증거 묶음에 연결되는지"가 보이게 하는 것입니다.

#### 소유권과 책임 배분

법무팀만 증거를 책임지는 구조는 오래 유지되지 않습니다. 법무팀은 기준을 만들고 검수하는 역할을 맡고, 실제 수집은 자료가 발생하는 곳에 두는 편이 현실적입니다.

| 폴더 | Owner | 수집 주기 | 법무 검수 주기 | 누락 시 생기는 문제 |
|---|---|---|---|---|
| 01_Sales | 마케팅팀 | 매월 | 분기 | 실제 노출 화면을 설명하지 못한다 |
| 02_Transaction | 운영팀 | 매월 | 분기 | 거래 흐름과 상업적 사용 설명이 약해진다 |
| 03_Packaging | 브랜드팀 | 변경 시 즉시 | 반기 | 등록 표기와 실제 사용 표기의 연결이 흔들린다 |
| 04_Advertising | 마케팅팀 | 캠페인 종료 후 1주 | 분기 | 사용 시점과 시장 노출 흔적이 비게 된다 |
| 05_Internal | 법무팀 | 이벤트 발생 시 | 수시 | 승인 경위와 내부 판단 근거가 사라진다 |
| 06_ThirdParty | 법무팀 | 계약 체결·갱신 시 | 수시 | 권리자와 실제 사용자의 연결 설명이 막힌다 |

Owner는 수집 책임자이지 법적 판단자가 아닙니다. 판단은 법무팀이 합니다. 이 구분을 분명히 해야 자료가 실제로 모입니다.

### 분기 점검과 예외 처리

증거 점검은 "자료가 있는가"보다 "지금 바로 제출 가능한 상태인가"를 보는 작업에 가깝습니다. 그래서 체크리스트만 남기는 것보다 점검 결과표를 남기는 편이 훨씬 유용합니다.

#### 분기 점검에서 남겨야 할 산출물

시장별로 아래 정도의 1페이지 점검표를 남기면 충분합니다.

| 시장 | 상태 | 누락 자료 | 보완 기한 | Owner |
|---|---|---|---|---|
| KR | 정상 | 없음 | — | 마케팅 / 운영 |
| US | 보완 | 최근 2개월 주문 증빙 부족 | 2025-04-15 | 운영팀 |
| MX | 공백 | 파트너 사용 권한 문서 미정리 | 2025-04-30 | 법무팀 |

상태값은 `정상`, `보완`, `공백` 정도로 단순하게 고정하는 편이 좋습니다. 상태를 단순화해야 보고도 빨라지고, 누락 자료의 우선순위도 더 쉽게 보입니다.

분기 점검 때 실제로 확인할 항목은 아래와 같습니다.

- 최근 3개월 자료가 각 시장 폴더에 들어와 있는가
- 파일명 규칙과 메타데이터가 유지되고 있는가
- 새로 추가된 판매 채널이 폴더 구조와 index 파일에 반영됐는가
- 브랜드 표기나 포장 변경이 `03_Packaging`과 `05_Internal`에 같이 남아 있는가
- 권리자와 실제 사용자가 다른 시장에서 연결 문서가 최신 상태인가
- 플랫폼 캡처의 URL과 날짜 증빙이 살아 있는가

점검 기록 자체가 "지속적으로 관리했다"는 운영 증거가 됩니다. 실제 분쟁 단계에서 자료 자체만큼 관리 이력도 설명력을 높여 줍니다.

#### 현실적인 예외와 실패 패턴

현업에서는 자료가 원칙대로 예쁘게 모이지 않는 경우가 더 많습니다. 그래서 예외를 막는 것보다, 예외가 생겨도 최소한의 보존 경로를 남기는 것이 중요합니다.

- **ERP export가 늦는 경우**: 월말마다 raw export 파일과 추출 일자를 먼저 보관하고, 정식 보고서는 나중에 덧붙입니다.
- **유통 파트너가 비협조적인 경우**: 계약서에 증거 협조 의무를 넣고, 그 전에는 정산서·입고확인·파트너 판매 화면 등 대체 자료를 같은 폴더에 모읍니다.
- **플랫폼 URL이 자주 바뀌는 경우**: 화면 캡처만 남기지 말고 URL, 캡처일, HTML 또는 PDF 저장본을 같이 보관합니다.
- **계열사 명의로 판매하는 경우**: owner-user linkage 메모와 계약 또는 내부 승인 문서를 `05_Internal`과 `06_ThirdParty`에 같이 남깁니다.

핵심은 완벽한 자료만 모으는 것이 아니라, 약한 자료라도 왜 약한지 알고 다음 보완 경로를 남기는 것입니다.

### 여러 국가에서 바로 재사용하려면

한 국가에서 만든 자료를 다른 국가 대응에 쓰려면, 문서 이름과 메타데이터보다 더 중요한 것이 "설명 방식의 일관성"입니다. 같은 브랜드라도 국가마다 요구하는 설득 포인트가 다르기 때문에, 공통 구조 위에 국가별 추가 메모만 얹는 방식이 가장 효율적입니다.

#### 국가별 추가 메모

- **미국**: 실제 상업적 거래 흐름의 설득력이 중요합니다. 판매 페이지, 주문, 배송 완료 자료를 같은 거래번호 또는 주문번호로 연결해 두면 설명이 훨씬 강해집니다.
- **중국**: 등록된 표기와 실제 사용 표기의 정합성이 민감합니다. 등록 표기 버전과 실제 사용 표기가 다르면 `03_Packaging`에 두 버전을 함께 두고, `05_Internal`에 승인 또는 변경 경위를 남기는 편이 안전합니다.
- **멕시코**: 파트너 사용과 사용선언 대비 자료 정리가 중요해질 수 있습니다. 유통 파트너 계약서, 사용 권한 문서, 최근 거래 자료를 같은 시장 폴더 안에서 바로 연결할 수 있어야 합니다.

국가별 대응은 별도 시스템을 하나씩 만드는 작업이 아니라, 공통 패킷에 국가별 설명 메모를 붙이는 작업에 가깝습니다. 그래서 공통 구조가 먼저 흔들리지 않아야 합니다.

### 지금 바로 시작할 수 있는 체크리스트

처음부터 모든 시장을 동시에 정비할 필요는 없습니다. 리스크가 빠르게 드러나는 2~3개 시장만 먼저 고르고, 그 시장에서 구조가 작동하는지 확인한 뒤 확장하는 편이 현실적입니다.

#### 이번 주 안에 할 것

2시간 안에 끝낼 수 있는 최소 시작점은 아래 정도입니다.

1. 분쟁 가능성이나 심사 일정 기준으로 우선 시장 2~3개를 고른다.
2. 위 폴더 구조를 그대로 만들고 각 시장 폴더에 `00_Index`를 만든다.
3. Owner, 주력 채널, 관련 계약 위치, 최근 업데이트일을 index 파일에 적는다.
4. 파일명 규칙 1페이지를 정리해 Owner에게 공유한다.

#### 이번 분기 안에 굳힐 것

- Owner별로 현재 가지고 있는 자료를 각 폴더로 이동한다.
- 빠진 자료 목록을 만든다. 없는 것보다 "무엇이 없는지 알고 있는 상태"가 더 중요하다.
- 법무팀이 각 시장 폴더를 한 번 검수하고, 파일명과 메타데이터 보완 요청을 남긴다.
- 분기 점검표 형식을 고정하고 `정상 / 보완 / 공백` 상태값을 통일한다.

#### 다음 분기부터 반복할 것

- `01_Sales`, `02_Transaction`, `04_Advertising`는 월 단위 수집 루틴을 고정한다.
- `03_Packaging`, `05_Internal`, `06_ThirdParty`는 변경 이벤트가 생길 때 즉시 업데이트한다.
- 분기마다 시장별 점검표를 남기고, 보완 항목은 기한과 owner까지 붙여 추적한다.
- 분쟁이나 플랫폼 신고가 발생하면 기존 폴더에서 증거 패킷을 바로 뽑아 별도 제출 폴더로 복제한다.

완벽한 자료를 한 번에 갖추는 것보다, 같은 구조로 계속 쌓이는 상태를 만드는 것이 우선입니다. 구조가 잡히면 담당자가 바뀌어도 이어지고, 국가별 대응도 훨씬 빨라집니다.
