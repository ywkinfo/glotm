# GloTm Harness Style Guide

이 문서는 루트 셸, 콘텐츠 스크립트, 원고 수정에 공통으로 적용되는 기본 작성 규칙을 정리한다.
워크스페이스별 세부 규격은 각 `Harness/Content-Spec.md`를 따른다.

## TypeScript

- `strict` 설정은 유지한다. 타입 오류를 문서로 정당화하지 않는다.
- `any`는 금지한다. 애매한 값은 좁은 유니온, 제네릭, 명시적 타입 가드로 해결한다.
- React 컴포넌트의 props는 `type` 또는 `interface`로 명시한다.
- 공용 유틸리티와 파이프라인 함수는 입력과 출력 타입이 읽히도록 유지한다.
- `noUncheckedIndexedAccess`를 우회하지 않는다. 인덱스 접근은 `undefined` 가능성을 처리한다.

## React And Runtime Code

- 컴포넌트는 단일 읽기 흐름 책임을 유지한다.
- 공용 동작은 `src/products/shared.ts`나 공용 컴포넌트로 모으고, 제품별 파일에는 제품 차이만 남긴다.
- 사용자 흐름에 직접 영향을 주는 라우팅, 검색, outline, continue reading 변경에는 회귀 테스트를 우선 붙인다.

## Markdown And Content

- 원고 파일의 첫 번째 비어 있지 않은 줄은 H1이어야 하며, H1은 한 번만 나온다.
- 조립형 문서에서는 문서 제목이 H1, 챕터 경계가 H2, 본문 섹션은 H3-H5를 사용한다.
- 표는 pipe table 문법을 유지하고, 코드 펜스는 항상 짝을 맞춘다.
- 내부 조사 마커, 브라우저 인용 표식, 임시 메모를 원고에 남기지 않는다.
- 문서 링크와 기관명은 독자가 추적 가능한 형태로 쓴다. 민감한 법률 사실은 공식 출처 기준으로 다시 확인한다.

## CSS

- 기존 패턴을 우선 따른다. 새로운 스타일 시스템을 병행 도입하지 않는다.
- 반복되는 색상, 간격, 크기는 토큰 또는 기존 변수에 맞춘다.
- 이유 없는 매직 넘버를 늘리지 않는다. 숫자가 필요하면 인접 규칙과의 관계가 읽히도록 둔다.
- 시각 변경은 모바일과 데스크톱의 읽기 흐름을 함께 고려한다.

## Tests And Verification

- 공용 로직, 검색 이동, 읽기 흐름 회귀는 가능한 한 `Vitest`로 고정한다.
- 테스트를 추가하지 못하는 변경은 이유가 명확해야 하며, 대신 수동 검증 경로를 남긴다.
- 빌드와 테스트 명령은 저장소의 실제 `package.json` 스크립트를 기준으로 쓴다.

## Commit Discipline

- 한 커밋은 한 논리적 변경만 담는다.
- 문서 구조 개편, 기능 변경, generated 산출물 갱신은 가능한 한 분리한다.

## Ratchet Rules

- 신규 파일과 대규모 리라이트 파일은 함수 50줄 이하, 파일 500줄 이하를 목표로 한다.
- 이미 큰 기존 파일은 이 규칙을 즉시 위반으로 보지 않는다.
- 다만 기존 대형 파일을 만질 때는 더 키우지 않는 방향을 우선하고, 큰 덩어리를 추가해야 한다면 분리 가능한지 먼저 본다.
